# Примеры творческих заданий по программированию (15 баллов)

## Описание

Это список идей творческих проектов, которые студенты могут использовать как вдохновение или как основу для своих идей. Каждый проект разработан так, чтобы соответствовать критериям оценивания на 15 баллов.

Студент может:
1. Выбрать один из предложенных проектов и реализовать его
2. Модифицировать проект под свои интересы
3. Придумать собственный проект, соответствующий критериям

---

## Категория 1: Образовательная аналитика и анализ данных

### Проект 1.1: "Learning Path Analyzer"

**Описание:** Система анализа пути обучения студента на основе логов LMS (Moodle, Canvas и т.д.). Анализирует, какие типы активностей наиболее эффективны для разных студентов.

**Ожидаемая полезность:** 4 балла

**Основной функционал:**
- Парсинг CSV файлов с логами LMS
- Выделение ключевых событий (login, assignment submission, forum posts, quiz attempts)
- Анализ корреляции между типами активностей и успеваемостью
- Визуализация результатов (matplotlib, plotly)
- Рекомендации для оптимизации пути обучения

**CI/CD идея:**
- Scheduled workflow для регулярного анализа новых логов
- Автоматическое создание отчета с графиками и коммит в ветку `reports/`
- Upload результатов как artifact

**README должен содержать:**
- Описание цели проекта
- Пример CSV входного формата
- Примеры вывода с графиками
- Пошаговую инструкцию по использованию

---

### Проект 1.2: "Assessment Question Difficulty Classifier"

**Описание:** Модель машинного обучения для автоматической классификации сложности вопросов теста/экзамена на основе текста вопроса, времени ответа и процента правильных ответов.

**Ожидаемая полезность:** 4 балла

**Основной функционал:**
- Dataset с вопросами и метриками сложности
- Feature extraction из текста (длина, количество слов, Flesch-Kincaid score)
- Train/test split на реальных данных
- Классификатор (Naive Bayes, SVM или Random Forest)
- Оценка моделей (confusion matrix, precision, recall)

**CI/CD идея:**
- При push на новую ветку автоматически переучить модель
- Запустить валидацию на отдельном тестовом наборе
- Сохранить метрики модели в JSON и закоммитить

## Категория 2: Обработка текста и NLP

### Проект 2.1: "Educational Plagiarism Detector"

**Описание:** Система для определения плагиата в студенческих работах, используя различные методы сравнения текстов (cosine similarity, sequence matching, n-grams).

**Ожидаемая полезность:** 4 балла

**Основной функционал:**
- Загрузка студенческих работ (txt, pdf файлы)
- Предварительная обработка (tokenization, lemmatization)
- Сравнение текстов несколькими методами:
  - Cosine similarity с TF-IDF
  - Longest common subsequence
  - N-gram matching
- Вывод процента похожести между работами
- Визуализация результатов в виде матрицы похожести

**CI/CD идея:**
- Trigger на загрузку файла в папку `uploads/`
- Автоматический запуск анализа
- Сохранение результатов в JSON с timestamp

---

### Проект 2.2: "Multilingual Learning Material Summarizer"

**Описание:** Инструмент для автоматического резюмирования учебных материалов с поддержкой нескольких языков (English, Russian, German).

**Ожидаемая полезность:** 4 балла

**Основной функционал:**
- Загрузка учебного материала (статьи, лекции в текстовом формате)
- Выбор языка текста
- Выбор уровня сжатия (20%, 30%, 50%)
- Абстрактивное резюмирование (использование transformers или простые методы)
- Вывод сжатого текста с ключевыми моментами

**CI/CD идея:**
- Поддержка нескольких версий Python для тестирования
- Автоматическое тестирование с разными языками

## Категория 3: Системы рекомендаций

### Проект 3.1: "Study Material Recommender"

**Описание:** Система рекомендации учебных материалов студентам на основе их предпочтений, уровня знаний и истории обучения.

**Ожидаемая полезность:** 4 балла

**Основной функционал:**
- Dataset с материалами и оценками студентов
- Основные подходы:
  - Collaborative filtering (item-item similarity)
  - Content-based filtering (анализ характеристик материала)
  - Гибридный подход
- Оценка качества рекомендаций (RMSE, precision@k)
- API для получения рекомендаций

**CI/CD идея:**
- Scheduled workflow для регулярного переобучения на новых оценках
- Artifact upload с метриками моделей

## Категория 4: Автоматизация оценивания

### Проект 4.1: "Code Quality Assessment Tool"

**Описание:** Инструмент для автоматического оценивания качества студенческого кода по набору метрик.

**Ожидаемая полезность:** 4 балла

**Основной функционал:**
- Анализ Python кода:
  - Соблюдение PEP 8 (используя AST парсинг)
  - Цикломатическая сложность функций
  - Покрытие тестами (coverage)
  - Наличие docstring
  - Дублирование кода (copy-paste detection)
- Выставление оценки на основе метрик
- Генерация отчета с рекомендациями

**CI/CD идея:**
- Workflow для проверки кода студентов из отдельной папки
- Автоматическое создание отчета и сохранение в artifact

---

### Проект 4.2: "Automated Essay Grading System"

**Описание:** Система для полуавтоматического оценивания эссе и письменных ответов на основе критериев.

**Ожидаемая полезность:** 4 балла (с использованием правильных метрик)

**Основной функционал:**
- Определение критериев оценивания (структура, грамматика, содержание, оригинальность)
- Анализ текста:
  - Flesch-Kincaid readability score
  - Количество грамматических ошибок (используя language_tool)
  - Проверка на плагиат
  - Анализ тональности (sentiment analysis)
- Выставление оценки и предоставление feedback

## Категория 5: Визуализация и аналитика

### Проект 5.1: "Interactive Learning Dashboard"

**Описание:** Веб-приложение для визуализации прогресса студентов с интерактивными графиками.

**Ожидаемая полезность:** 3-4 балла

**Основной функционал:**
- Backend (Python Flask/FastAPI):
  - API для получения данных о прогрессе студентов
  - Расчет метрик (средний балл, тренд улучшения и т.д.)
- Frontend (Streamlit или plotly):
  - Интерактивные графики (Plotly)
  - Фильтры по студентам, предметам, периодам
  - Сравнение студентов/групп
- Экспорт данных в PDF/CSV

**CI/CD идея:**
- Развертывание на GitHub Pages или Streamlit Cloud через workflow_dispatch
- Автоматическое обновление Dashboard на регулярной основе (schedule)

## Категория 6: Инструменты для преподавателей

### Проект 6.1: "Test Case Generator"

**Описание:** Инструмент для автоматического генерирования тестовых случаев для различных типов заданий по программированию.

**Ожидаемая полезность:** 3-4 балла

**Основной функционал:**
- Шаблоны для разных типов задач (sorting, searching, data structures)
- Генератор случайных входных данных
- Создание edge cases (пустые списки, очень большие числа)
- Генерация ожидаемых выходов
- Экспорт в JSON формат для использования в тестовых фреймворках

**CI/CD идея:**
- Trigger на workflow_dispatch для интерактивного запуска генератора
- Результаты сохраняются в artifact

---

### Проект 6.2: "Assignment Variation Generator"

**Описание:** Система для создания уникальных вариантов заданий для студентов, чтобы избежать плагиата.

**Ожидаемая полезность:** 4 балла

**Основной функционал:**
- Шаблон задания с параметрами (например, числовые диапазоны, названия переменных)
- Генератор случайных вариантов
- Сохранение mapping между студентом и его вариантом
- Возможность регенерации для конкретного студента
- Экспорт вариантов в PDF или Markdown

**CI/CD идея:**
- Scheduled workflow для ежедневного создания вариантов и их распределения студентам
- Интеграция с Moodle через API

## Как выбрать проект?

### Для получения максимума баллов (13-15):

1. **Выберите проект из категорий 1-4** — они специально разработаны для получения 4 баллов за полезность
2. **Убедитесь, что:**
   - Структура папок логична (3 балла за оформление)
   - Код работает, есть тесты, PEP8 проверка (4 балла)
   - README хороший с примерами (2 балла)
   - CI/CD workflow делает что-то полезное, не только проверку кода (2 балла)

### Для получения хороших баллов (10-12):

- Можно выбрать более простой проект, но хорошо его реализовать
- Особое внимание на CI/CD — это часто упускается студентами

### Типичные ошибки:

- ❌ Выбирают слишком простой проект (TODO app, Calculator)
- ❌ Забывают про CI/CD automation (только linting, без запуска тестов или artifact upload)
- ❌ Плохой README (без примеров использования)
- ❌ Беспорядок в репозитории (код в корне, нет папок)
- ❌ Не запускают тесты локально перед push (CI/CD не работает)

## Рекомендуемый процесс разработки

1. **Неделя 1:** Выбрать проект, создать репо, подготовить структуру, написать README
2. **Неделя 2:** Реализовать основной функционал, написать тесты
3. **Неделя 3:** Настроить GitHub Actions workflow, добавить креативный элемент (schedule/artifact/dispatch)
4. **Неделя 4:** Финальное тестирование, оптимизация README, проверка всех критериев
